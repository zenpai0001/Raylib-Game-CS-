Making the raylib game in C#. Raylib_cs.
Using vector2 rather than int x and y to make it easier to move the player and other objects around.

Also discovered that I can make an object to call another object in the same class. 
This'll likely be useful for later on. Example: Instantiaing a game object in static void main
and then calling the player object from the game object to move it around and do other things.
Essentially, it is inheritance without actually forcing a class to inherit from another class. It is more of a composition approach, which is what I prefer.

Now imagine making a hybrid approach where I have a base class called GameObject and then have the player class inherit from that base class.
Back and forth between composition and inheritance to design an ECS for the game.

When it comes to shared logic and contracts within the same class, I can use interfaces.
For example. I can make an interface called IPlayer and then have the player class implement that interface
or IDamageable and have the player class implement that interface as well. The equivalent to using header files in C++.

Take note:
Interfaces are contracts that a class must follow. They define a set of methods and properties that a class must implement, but they do not provide any implementation themselves. 
This allows for a lot of flexibility in how classes can be designed and structured, while still ensuring that they adhere to certain standards and expectations.
Shared logic and contracts within the same class can be achieved through the use of interfaces, which allow for a clear separation of concerns and promote code reusability.
